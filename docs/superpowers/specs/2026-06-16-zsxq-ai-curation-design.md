# 知识星球 AI 精选筛选层 — 设计文档

- 日期:2026-06-16
- 项目:zsxq-clipper(Chrome MV3 扩展)
- 状态:待用户审阅

## 1. 背景与目标

现状:zsxq-clipper 已能在 `wx.zsxq.com` 帖子上点「📋 剪藏」逐篇保存为 Markdown。用户**刻意**采用"点按才保存"的方式——目的是主动筛选、防止噪音自动入库。

**核心痛点**:信息过载。星球帖子太多,用户"根本不知道内容是什么",缺乏一个**前置筛选**环节,无法在浏览当下低成本判断价值。

**目标**:在现有剪藏流程之上,增加一个 **AI 前置筛选层**——在用户决定收藏之前,先用"质量信号 + AI 逐条评分"把一个星球今日的高价值帖子筛出来,呈现在侧边精选栏。**最终收藏仍由用户手动点按**,绝不自动入库(保留主动筛选意图)。

**非目标**:
- 不做全自动抓取/入库(用户明确拒绝)
- 不做个性化推荐画像系统(维护重、易不准)
- 不做跨星球聚合(本期只做"当前星球·今日")

## 2. 核心设计决策(对话已确认)

| 维度 | 决策 |
|------|------|
| 筛选依据 | 质量信号**粗筛** + AI **逐条精评**(给分+一句理由) |
| 呈现形态 | 右侧**精选栏**,只列高分帖,点 `[收]` 直接收、点标题跳原帖 |
| 覆盖范围 | 当前星球 · 今日全部帖子 |
| 技术方案 | **纯扩展**(不依赖本地 server) |
| AI key | 存 `chrome.storage.local`,用户在设置里填(**不硬编码进源码**,因项目已开源) |
| 落盘方式 | **File System Access API**,授权写入 `~/zsxq-clips/`,与现有 wiki 入库链衔接 |

## 3. 架构与数据流

```
打开 wx.zsxq.com 某星球
   │  content.js 注入右侧「精选栏」+ 「🔍 扫描今日」按钮
   ▼ 点击扫描
background.js (service worker)
   ① fetch 星球「今日帖子」API(复用登录 cookie;host_permissions 已覆盖 *.zsxq.com)
   ② 取结构化列表(标题/正文摘要/作者/点赞/评论/精华)→ 本地算质量信号粗筛
   ③ 对粗筛后候选逐条调 AI → 输出 {score, reason, tags}
   ▼ 返回评分列表
content.js 渲染精选栏:高分在上、低分折叠
   ▼ 用户点 [收]
content.js 用 File System Access API 写入授权的 ~/zsxq-clips/ (+ assets/),复用现有「单篇剪藏」的提取/外链抓取/图片本地化逻辑
   → 落 ~/zsxq-clips/ 顶层 = 待入库 SOURCE,衔接现有 wiki 入库流程
```

**组件改动**:
- `content.js`:新增精选栏 UI + 扫描触发 + 落盘改走 File System Access(取代现有 `saveToServer`)
- `background.js`:新增「拉星球 API」+「调 AI 评分」两个消息处理
- 新增 `settings`(popup 或选项页):填 API key + 关注领域 + 首次授权目录
- `server.js`:**不再是必需**。本期落盘统一走 File System Access;`server.js` 可保留作可选 fallback,但默认不依赖

## 4. 评分管线(核心)

### 4.1 粗筛层(本地、免费、快)

对「今日全部」帖子(可能上百条)计算质量信号:
- 点赞数、评论数
- 是否精华
- 信息密度:正文字数、是否含代码块、链接数、图片数

按阈值砍掉低质,例如:`点赞 < 3 且 非精华 且 字数 < 100` → 丢弃。阈值可在设置里调。目标:把候选砍到 20–40 条,交给 AI。

### 4.2 AI 精评层(调 GLM)

对粗筛后候选逐条送 AI,输入帖子摘要 + 元数据,prompt 要求返回 JSON:
```json
{ "score": 8, "reason": "一句中文理由", "tags": ["RAG", "实战"] }
```

- score:1–10,"对该用户的有用程度"
- reason:一句话,显示在精选栏
- tags:抽取的主题标签

AI 返回需**容错解析**(模型偶尔返回非合法 JSON 或多余文本,用正则/兜底提取)。

## 5. AI 评分标准与成本控制(待 review 确认)

- **评分标准**:默认通用信息质量/实用性(信息密度、是否可操作、是否有独特观点)。叠加用户在设置里填的一段轻量**关注领域**(如"AI 编程、独立开发、RAG"),作为 prompt 的一部分。**不做画像系统**,就是一段可改的文本。
- **模型**:GLM(用户有 Coding Plan,便宜;评分任务用 glm-4.7/flash 级即可)。
- **成本控制**:
  - 粗筛后才调 AI(量已砍到几十条)
  - 按 `topic_id + 内容hash` 缓存评分结果,同一天不重复评
  - 单次扫描 AI 调用设上限(默认 ≤ 50 条),超出只展示已评的并提示

## 6. 落盘:File System Access API

- 首次使用:在设置页/精选栏调 `showDirectoryPicker()`,用户授权 `~/zsxq-clips/`,DirectoryHandle 存 IndexedDB 持久化
- 保存 `.md`:用授权句柄直接写文件到该目录(顶层 = 待入库 SOURCE)
- 图片本地化:写到 `assets/<slug>/`,Markdown 内图片链接替换为相对路径(沿用现有逻辑)
- **体验代价**:浏览器重启后第一次保存,需用户点"允许访问该目录"(本地 server 是完全无感的)。这是换取"零 server 依赖 + 便携"的代价
- `chrome.downloads` 不采用(目录受限到 Downloads,与入库目录不一致)

## 7. API key 管理

- 用户在扩展设置里填 key,存 `chrome.storage.local`
- 源码只含"读取 key 调用 AI"的逻辑,**不含任何 key 值**(项目已开源到 GitHub,硬编码 = 泄露)
- 每个使用者填自己的 key

## 8. 精选栏 UI 与交互

右侧固定栏:
- 顶部:「🔍 扫描今日」按钮 + 过滤(只看精华/全部)+ 授权目录状态
- 列表每条卡片:`⭐分数 · 标题 / 作者·❤️点赞·💬评论 / 一句 AI 理由 / [📋 收][🔗 跳原文]`
- 按 score 降序;低分(如 < 6)折叠到底部"更多"区
- `[收]` 触发现有单篇剪藏流程(含外链抓取 + 图片本地化),落盘走 File System Access

## 9. 错误处理与降级

- **星球 API 拉取失败/登录过期**:精选栏提示"请重新登录知识星球",不崩
- **AI 评分超时/失败**:降级为**只用质量信号排序**,该条标注"AI 未评",不阻断
- **AI 返回非法 JSON**:容错解析,解析失败则该条不显示理由、仅按质量信号排序
- **未授权目录 / File System Access 不可用**:点 `[收]` 时提示去设置授权;可降级用 `chrome.downloads` 兜底
- **跨天**:缓存按日期失效,自动重评
- **粗筛后 0 候选**:提示"今日无值得精筛的内容"

## 10. 测试策略

- **单元**:
  - 质量信号计算正确性
  - 粗筛阈值边界
  - AI 返回 JSON 的解析与容错(含非法/多余文本用例)
  - File System Access 句柄的持久化与权限重验
- **集成**:mock 星球 API 响应 → 跑完整「拉取→粗筛→评分→渲染」管线
- **评分质量回归**(亮点):用 `~/zsxq-clips/.archived/` 中**已入库帖子**作正样本,验证评分模型对它们打高分——复用用户的入库数据做基准

## 11. 已知风险与未决项

1. **星球拉帖 API 结构待实测**(实现期标红):端点路径、分页参数、按"今日"过滤方式、字段结构都需打开页面抓接口确认。方向可行(cookie + host 权限齐备),但具体 API 是 known-unknown。
2. **File System Access 重启确认**:每次浏览器重启首写需点"允许"。是否可接受,以用户实际体验为准。
3. **AI 评分标准 / 触发方式 / 基准复用**(第 5、8 节相关):为 brainstorming 中的建议性决策,**待用户 review 确认**:
   - AI 标准 = 通用质量 + 轻量关注领域文本(非画像)——OK?
   - 触发 = 手动点「扫描今日」(用户控成本),不自动——OK?
   - 用已入库帖子做正样本回归——认可?
4. **MV3 service worker 生命周期**:批量调 AI 可能耗时较长,需注意 worker 不会被中断(必要时分批 + 消息续传)。
